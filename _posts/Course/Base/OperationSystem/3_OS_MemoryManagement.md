## 2. 实现机制
### 2.1 地址转换
#### 页表(Page Table)
页表是存储虚拟页到物理页帧映射关系的数据结构

#### 二级页表
将页表分级,减少页表占用的内存空间
#### 多级页表

扩展二级页表的概念,进一步减少内存占用(如x86_64使用四级页表)

#### 页表项(PTE)的结构
物理页帧号(PFN)
存在位(Present bit):指示该页是否在内存中
读写权限位(Read/Write bits)
用户/超级用户位(User/Supervisor bit)
修改位(Dirty bit):指示该页是否被写入
访问位(Accessed bit):指示该页是否被访问过
其他位:如缓存禁用位、全局位等

### 2.2 硬件支持
#### 内存管理单元(MMU)
1. MMU(Memory Management Unit,内存管理单元) 是 CPU 中的一个硬件组件,负责:
2. 虚拟地址到物理地址的转换
3. 内存访问权限检查
4. 触发缺页异常
#### 快表(TLB,Translation Look aside Buffer)
是MMU的一部分,用于缓存页表项,当TLB未命中时,需要查找页表,并更新TLB


### 2.3 异常处理
缺页异常是当程序访问一个虚拟内存地址,但该地址对应的物理页面尚未加载到内存时,由 MMU 触发的一种软中断

#### 缺页异常(Page Fault)的类型
硬缺页(Hard Page Fault):所需页面不在内存中,需要从磁盘调入
软缺页(Soft Page Fault):页面在内存中,但未在当前进程的页表中建立映射(如共享内存)
无效缺页(Invalid Page Fault):访问的地址无效(如越界访问),会导致段错误


#### 处理流程

1. 操作系统捕获缺页异常,保存现场
2. 检查发生缺页的虚拟地址是否合法(在进程的地址空间内)以及访问权限
3. 如果合法,则分配一个物理页帧,并将所需页面从磁盘(或共享内存等)加载到该页帧
4. 更新页表,建立虚拟页到物理页帧的映射
5. 重新执行引发缺页的指令



### 2.4 文件执行流程
1. 文件加载阶段
   1. Shell解析命令:Shell解析 ./test,发现是一个可执行文件
   2. 系统调用:Shell通过 execve() 系统调用请求内核执行该程序
   3. 文件系统查找:内核在文件系统中查找 ./test 文件
   4. 读取文件头:从硬盘读取可执行文件的头部信息(ELF格式)
   5. 权限检查:检查文件是否有执行权限
2. 内存映射和页表建立
   1. 分配虚拟地址空间:内核为进程分配虚拟地址空间
   2. 创建页表:建立进程的初始页表结构(多级页表)
   3. 映射代码段:将可执行文件的代码段映射到虚拟内存
   4. 映射数据段:映射数据段、BSS段等
   5. 设置堆栈:分配堆和栈空间,并建立相应的页表项
3. 程序开始执行 - 地址翻译过程
   1. 第一次内存访问(取指令):
      1. CPU生成指令的虚拟地址
      2. 查询TLB(在CPU中)→ 未命中(因为刚开始执行)
      3. 查询多级页表(访问内存条/DRAM)
      4. 页表返回物理地址
      5. 用物理地址访问内存条,读取指令数据
      6. 同时将这次翻译缓存到TLB中
   2. 后续内存访问:
      1. 如果访问的地址在TLB中命中:直接从TLB获取物理地址,一步访问内存
      2. 如果未命中:重复上述多级页表查询过程
4. 缺页异常处理
   1. 触发缺页异常:MMU发现页表项标记为"不在内存中"
   2. 页面调入:操作系统从硬盘加载需要的页面到物理内存
   3. 更新页表:修改页表项,标记页面已在内存中
   4. 重新执行:CPU重新执行引发异常的指令


### 2.5 流程中使用到的软硬件
| 组件/步骤    | 实现方    | 说明                                     |
| :----------- | :-------- | :--------------------------------------- |
| MMU / TLB    | 硬件      | CPU内部的专用电路                        |
| 页表遍历     | 硬件      | MMU自动完成,软件不可见                   |
| 触发缺页异常 | 硬件      | MMU在遇到无效页表项时自动发出信号        |
| CR3寄存器    | 硬件      | 但由OS在进程切换时设置其值               |
| 多级页表结构 | Linux内核 | 内核代码定义并管理页表在内存中的数据结构 |
| 缺页处理程序 | Linux内核 | 分析原因、决定如何处理的C代码函数        |
| 页面置换算法 | Linux内核 | 决定换出哪个页面的策略代码               |
| 磁盘I/O      | Linux内核 | 调用驱动,从交换区或文件系统读/写页面数据 |
| 更新页表     | Linux内核 | 在处理缺页后,内核代码修改页表项内容      |
| 刷新TLB      | 协同      | 内核执行特定指令,由硬件完成实际刷新      |



### 2.6 malloc流程

整个过程可以分为三个阶段:1. 虚拟地址分配(纯软件) -> 2. 物理内存延迟分配(硬件触发软件) -> 3. 物理内存绑定(软件响应硬件)

#### 阶段一:纯软件 - 虚拟地址预留 (malloc)

1. 用户程序调用 malloc(size)
2. Glibc 库函数接管:
      malloc 是C标准库(通常是Glibc)提供的函数,它不是系统调用
      Glibc 会先在自己管理的堆内存池中寻找一块足够大小的空闲虚拟内存这个内存池是Glibc通过系统调用预先从操作系统那里"批发"来的
      如果内存池中没有足够空间,Glibc 会通过 brk 或 sbrk 系统调用来调整程序的 program break(堆的顶端),以扩大堆空间；或者对于大块内存,使用 mmap 系统调用 在堆和栈之间创建一个新的内存映射区域
3. 操作系统的角色:
      内核响应 brk/mmap 系统调用,更新进程的内存描述符(mm_struct),标记出一段新的虚拟地址范围(VA)为"合法可用于堆或内存映射区"
      关键点:此时,操作系统只是修改了软件数据结构,承诺这段虚拟地址未来可以使用,但并没有分配任何物理内存,也没有修改页表对应页表的PTE(页表项)是空的/无效的
4. malloc 返回:Glibc 记录这块内存已被分配,然后将一个虚拟地址(VA) 返回给用户程序


#### 阶段二:硬件触发 - 缺页异常(第一次访问)

1.  程序拿到 malloc 返回的指针,第一次对其进行读写访问(例如 ptr = 'a')
2.  CPU 执行这条指令,需要将数据写入指针指向的虚拟地址(VA)
3.  MMU 介入,尝试将 VA 翻译为物理地址(PA)流程如下:
       查 TLB,未命中(因为是第一次访问)
       查 多级页表(在DRAM中)
       发现对应的页表项(PTE) 是无效的(Present位为0)
4.  MMU 触发缺页异常,CPU 中断当前执行流,切换到内核模式,跳转到Linux的缺页中断处理程序


#### 阶段三:软件响应 - 物理内存绑定(缺页处理)

1.  Linux缺页处理程序 (do_page_fault) 开始工作:
       它检查出错的虚拟地址,发现这个地址位于堆区(是之前通过 brk/mmap 合法分配的)
2.  内核分配物理内存:
       内核从系统的物理页帧空闲链表中,取出一页(例如4KB)空闲的物理内存
3.  更新页表:
       内核修改当前进程的页表,将出错的虚拟地址映射到刚刚分配的物理页帧上,并将PTE标记为有效,设置好读写权限
4.  返回并重试:
       缺页处理程序结束,CPU返回用户模式,并重新执行那条触发异常的指令(ptr = 'a')
5.  成功访问:
       这次再执行,MMU翻译虚拟地址:
           查TLB(可能还未缓存,但没关系)
           查页表,现在PTE是有效的了,成功获得物理地址
           数据被成功写入物理内存
       同时,这个映射关系会被加载到TLB中,加速后续的访问

## 3. 管理策略
#### 3.1 页面分配策略
固定分配:进程在创建时确定分配多少物理页面
可变分配:根据进程的运行情况动态调整分配的页面数
### 3.2 页面置换策略
最佳置换(OPT):置换未来最长时间内不再被访问的页面,理论上最优,但无法实现
先进先出(FIFO):置换最早进入内存的页面,可能产生Belady异常(分配页面数增加但缺页率反而上升)
最近最少使用(LRU):置换最近最长时间未被访问的页面,接近OPT,但实现开销大
时钟算法(Clock):近似LRU,使用一个环形链表和访问位,扫描时检查访问位,如果为1则清零并跳过,为0则置换
最不常用算法(LFU)
### 3.3 工作集模型
工作集:一个进程在时间区间Δ内访问的页面集合
基于工作集的页面置换算法:确保进程的工作集在内存中,以减少缺页
